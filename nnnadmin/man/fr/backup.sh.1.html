NAME

backup.sh - gère l'importation et l'exportation des données depuis des modules

 

SYNOPSIS

backup.sh [options] -l

backup.sh [options] <module> import|export <destination>

 

DESCRIPTION

backup.sh peut gérer les données sauvegardées d'un ensemble de modules en exportant (sauvegardes) et en important (restauration) des données.

Avec ce script vous pouvez gérer facilement une application qui utilise de multiples modules.

 

OPTIONS

-D
indique qu'il faut efface les données actuelles du module avant l'importation.

-M <répertoire>
indique l'emplacement des fichiers de modules (voir MODULES). défaut : backup-modules

-E <fichier>
indique l'emplacement du fichier d'environnement (voir MODULES). défaut : aucun

-l
affiche la liste des modules.

-A nosys
l'action est effectuée pour tous les modules qui n'ont pas le préfixe sys. A utiliser à la place de <module>.

-A allsys
l'action est effectuée sur tous les modules. A utiliser à la place de <module>.

-v
affiche la version du programme backup.sh.

-?
affiche le message d'aide.

 

COMMANDES

export <répertoire>
exporte les données du module dans le répertoire.

import <répertoire>
importe les données du module depuis les données sauvegardées dans le répertoire.

 

MODULES

Un module correspond à un fichier qui décrit comment exporter ou importer les données d'un composant. C'est un fichier script avec un format particulier (voir ci-dessous). Ce fichier contient des fonctions shell et une description. Un fichier d'environnement peut être utilisé pour conserver des paramètres ou des fonctions globales.

 

Fichier d'environnement

C'est un script optionnel lancé dans le shell courant (avec la commande . (point) ). Ce fichier doit être indiqué avec l'option -E
 .

 

Nom du fichier

Le fichier module doit présent dans un répertoire (par défaut backup-modules ) en suivant un de ces formats : <nombre>_<nom> ou <nombre>_sys_<nom> où
<nombre>
est un nombre. Il indique l'ordre d'exportation du module quand l'option -A est utilisée. Lors de l'importation l'ordre est inversé.
<nom>
est le nom que vous voulez.
sys
est utilisé pour indiquer un module système.

 

Modules Systèmes

Un module système représente un composant utilisé globalement sur la machine, comme un serveur ldap ou un serveur dns c'est à dire quand l'arrêt du composant a un impact sur l'ensemble de la machine et des autres applications.

 

Format du fichier

Le fichier doit contenir la description du module (commentaires avec NNNA) et 2 fonctions shells :
fn_export
fn_import

Ci-dessous, un modèle pour les fichiers :


 #!/bin/sh


 # ################################################# #
 # NNNA description décrire ici le composant         #
 # NNNA author      vous                             #
 # NNNA version     version de ce script             #
 # ################################################# #


 fn_import()
 {
  if [ "$1" = '-D' ]; then
   #détruit toutes les donnes (effacement)
   shift 1
   foobar
  fi
  #importe les données depuis $1
  foo $1
  return 0 #ok
 }


 fn_export()
 {
   #exporte les données vers $1
   faabar $1
   return 0 #ok
 }

 

EXEMPLE

Par exemple, si votre application est composé d'un site web et d'une base de données, vous pouvez créer deux modules :
un pour le site web,
un autre pour la base de données.

 

Site web

Supposons que le site web soit hébergé dans un virtualhost sur un Apache mutualisé. L'Apache est donc un composant système. Nous allons créer le fichier backup-modules/20_sys_apache. Ci-dessous son contenu:
 #!/bin/sh


 # ################################################# #
 # NNNA description Serveur web apache               #
 # NNNA author      Auteur de l'exemple              #
 # NNNA version     version 1.0                      #
 # ################################################# #


 fn_export()
 {
  tar czvf $1/apache.tgz /var/nnnadminexample/mywebsite/htdocs
  return $?
 }
 
 fn_import()
 {
  if [ "$1" = '-D' ]; then
   #la suppression avant l'import n'est pas supportée
   shift 1
  fi
  cd /
  tar xzf $1/apache.tgz 
  cd ->/dev/null
  return $?
 }

 

Base de données

L'instance de la base de données est lancée en mode autonome mais doit être lancée avant l'Apache. Nous allons créer le fichier backup-modules/10_database. Ci-dessous son contenu
 #!/bin/sh


 # ################################################# #
 # NNNA description Base de données du site web      #
 # NNNA author      Auteur de l'exemple              #
 # NNNA version     version 1.0                      #
 # ################################################# #
 


 PG_DIR=/usr/lib/postgresql/9.1/bin
 PGDATA=/var/nnnadminexample/mydatabase
 
 fn_export()
 {
  su - postgres -c "PGDATA=$PGDATA;$PG_DIR/pg_dump -Ft -f $1/base.tar"
  return $?
 } 
 
 fn_import()
 {
  unset opt
  if [ "$1" = '-D' ]; then
   opt='-c'
  fi
  su - postgres -c "PGDATA=$PGDATA;$PG_DIR/pg_restore $opt -Ft <$1/base.tar"
  return $?
 }

 

Exporter la base de données seulement (dans le répertoire /backup)

backup.sh database export /backup

 

Exporter tous les modules

backup.sh -A allsys export /backup

 

Importer les données d'apache depuis le répertoire /backup

backup.sh apache import /backup
