NAME

admin.sh - gestion du lancement et de l'arret de modules

 

SYNOPSIS

admin.sh [options] -l

admin.sh [options] <module> start|stop|status

admin.sh [options] <module> restart|force-reload

admin.sh [options] <module> try-restart

 

DESCRIPTION

admin.sh est l'équivalent des scripts d'initialisation LSB (Linux Standard Base) mais a un niveau applicatif. Il peut gérer un ensemble de modules en lançant, arrêtant ou en affichant l'état des modules.

Avec ce script vous pouvez gérer facilement une application qui utilise de multiples modules.

 

OPTIONS

-M <répertoire>
indique l'emplacement des fichiers de modules (voir MODULES). défaut : admin-modules

-E <fichier>
indique l'emplacement du fichier d'environnement (voir MODULES). défaut : aucun

-l
: affiche la liste des modules.

-A nosys
l'action est effectuée pour tous les modules qui n'ont pas le préfixe sys. A utiliser à la place de <module>.

-A allsys
l'action est effectuée sur tous les modules. A utiliser à la place de <module>.

-v
affiche la version du programme admin.sh.

-?
affiche le message d'aide.

 

COMMANDES

start
: démarre un module.

stop
: arrête un module.

status
: affiche l'état d'un module.

restart
: arrête le module (si démarré) puis redémarre le module. force-reload est pour correspondre avec la norme LSB.

try-restart
: redémarre le module (si démarré)

 

MODULES

Un module correspond à un fichier qui décrit comment démarrer, arrêter ou connaître l'état d'un composant. C'est un fichier script avec un format particulier (voir ci-dessous.) Ce fichier contient des fonctions shell et une description. Un fichier d'environnement peut être utilisé pour conserver des paramètres ou des fonctions globales.

 

Fichier d'environnement

C'est un script optionnel lancé dans le shell courant (avec la commande . (point) ). Ce fichier doit être indiqué avec l'option -E
 .

 

Nom du fichier

Le fichier module doit présent dans un répertoire (par défaut admin-modules ) en suivant un de ces formats : <nombre>_<nom> ou <nombre>_sys_<nom> où
<nombre>
est un nombre. Il indique l'ordre de lancement du module quand l'option -A est utilisée. Lors de l'arrêt l'ordre est inversé.
<nom>
est le nom que vous voulez.
sys
est utilisé pour indiquer un module système.

 

Modules Systèmes

Un module système représente un composant utilisé globalement sur la machine, comme un serveur ldap ou un serveur dns c'est à dire quand l'arrêt du composant a un impact sur l'ensemble de la machine et des autres applications.

 

Format du fichier

Le fichier doit contenir la description du module (commentaires avec NNNA) et 3 fonctions shells :
fn_status
fn_start
fn_stop

Ci-dessous, un modèle pour les fichiers :


 #!/bin/sh


 # ################################################# #
 # NNNA description décrire ici le composant         #
 # NNNA author      vous                             #
 # NNNA version     version de ce script             #
 # ################################################# #


 fn_status()
 {
  #récupere l'état du composant (comme dans System V)
  status=0 #0 le composant est démarré
  status=1 #1 le composant est mort mais le fichier PID existe
  status=2 #2 le composant est mort mais le fichier verrou existe
  status=3 #3 le composant est arrêté
  return $status
 }


 fn_start()
 {
  #démarre le composant
  return 0 #le composant est démarré
  #autre code de sortie = erreur
 }


 fn_stop()
 {
  #arrête le composant
  return 0 #le composant est arrêté
  #autre code de sortie = erreur
 }

 

EXEMPLE

Par exemple, si votre application se compose d'un site web et d'une base de données, vous pouvez créer deux modules :
un pour le site web,
un autre pour la base de données.

 

Site web

Supposons que le site web soit hébergé dans un virtualhost sur un Apache mutualisé. L'Apache est donc un composant système. Nous allons créer le fichier admin-modules/20_sys_apache. Ci-dessous son contenu:
 #!/bin/sh


 # ################################################# #
 # NNNA description Serveur web apache               #
 # NNNA author      Auteur de l'exemple              #
 # NNNA version     version 1.0                      #
 # ################################################# #


 fn_status()
 {
  service apache2 status
  return $?
 }


 fn_start()
 {
  service apache2 start
  return $?
 }


 fn_stop()
 {
  service apache2 stop
  return $?
 }

 

Base de données

L'instance de la base de données est lancée en mode autonome mais doit être lancée avant l'Apache. Nous allons créer le fichier admin-modules/10_database. Ci-dessous son contenu
 #!/bin/sh


 # ################################################# #
 # NNNA description Base de données du site web      #
 # NNNA author      Auteur de l'exemple              #
 # NNNA version     version 1.0                      #
 # ################################################# #
 
 PG_CTL=/usr/lib/postgresql/9.1/bin/pg_ctl
 
 fn_status()
 {
  su - postgres -c "$PG_CTL status -D /var/nnnadminexample/mydatabase"
  case $? in  #transform the pg_ctl exit status on system V
  '0') return 0;;
  '1') return 3;;
  *) return 4;;
  esac
 }
 
 fn_start()
 {
  su - postgres -c "$PG_CTL start -D /var/nnnadminexample/mydatabase"
  return $?
 }


 fn_stop()
 {
  su - postgres -c "$PG_CTL stop  -D /var/nnnadminexample/mydatabase"
  return $?
 }

 

Lancement de la base de données seulement.

admin.sh database start

 

Affiche l'état de tous les composants.

admin.sh -A allsys status

 

Arrête les modules qui ne sont pas système.

admin.sh -A nosys stop

 

Relance l'Apache

admin.sh apache restart
